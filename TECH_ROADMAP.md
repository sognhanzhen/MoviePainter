# MoviePainter 技术路线图

本文档定义 MoviePainter 当前阶段的技术推进顺序，用于指导从现有全栈 starter 逐步演进到以 Supabase 为数据管理底座的完整产品。

## 一、当前技术基线

当前仓库已经具备：

- React + TypeScript + Vite 前端
- Express + TypeScript 后端
- SQLite 数据库
- JWT + bcrypt 认证能力

但当前已经确认的目标技术方向是：

- 后台数据统一由 Supabase 管理
- 官方精选海报库统一由 Supabase 管理
- 用户所有业务数据统一由 Supabase 管理

推荐进一步收拢为：

- Supabase Postgres 负责结构化数据
- Supabase Storage 负责图片与资源文件
- Supabase Studio 负责当前阶段的后台数据维护
- Supabase Auth 作为推荐的下一阶段统一身份方案

## 二、技术推进原则

- 先建立 Supabase 数据底座，再扩展用户侧业务功能
- 先做用户侧页面闭环，再决定是否需要自建后台管理页
- 先做轻量状态机，再做复杂工作流引擎
- 先做结构化数据建模，再做高级推荐与智能化
- 让 Express 从“通用 CRUD 后端”逐步收敛为“编排 / AI 集成 / 安全代理层”

- 固化架构方向
- 固化 Supabase 技术路线

### Phase 1：Supabase 基础设施落地

目标：

- 先建立统一的数据与内容管理底座

重点工作：

- 创建 Supabase 项目
- 配置 Postgres 数据库
- 配置 Storage buckets
- 定义首批业务表结构
- 配置基础访问策略与权限策略
- 明确 Supabase Studio 作为当前后台管理入口

完成标志：

- 海报库、用户数据、生成记录已有统一存储位置
- 团队可以通过 Supabase Studio 维护数据

### Phase 2：应用骨架与路由层

目标：

- 从当前登录首页过渡为真正的多页面应用结构

重点工作：

- 建立公共路由与受保护路由
- 增加 landing / login / register / workspace / library / history / settings 页面壳层
- 增加用户登录后跳转到 `/workspace`
- 接入应用内基础会话判断

完成标志：

- 所有核心用户页面都能进入
- 路由守卫生效
- 基础导航结构建立完成

### Phase 3：生成工作区基础版

目标：

- 先做出可交互的核心工作台

重点工作：

- 实现 `AI Chat / AI Draw` 模式切换
- 实现工作区上半区与下半区布局
- 实现下方不规则灵感区布局
- 实现工作区内灵感海报注入逻辑
- 引入轻量状态机，跟踪当前模式、选中海报与生成状态

完成标志：

- `AI Chat` 模式可用
- `AI Draw` 模式壳层可用
- 灵感海报进入当前工作区逻辑跑通

### Phase 4：海报库页闭环

目标：

- 打通“海报库浏览 -> 详情 -> Use -> 模式选择 -> 工作区”的完整链路

重点工作：

- 从 Supabase 读取官方精选海报列表
- 实现顶部筛选模块
- 实现海报详情弹窗
- 实现 `Use` 后选择 `AI Chat / AI Draw`
- 跳转到目标工作区并带入海报参考数据

完成标志：

- 海报库闭环完成
- 详情弹窗与工作区联动完成

### Phase 5：历史记录与个人设置

目标：

- 建立用户中心能力

重点工作：

- 实现历史生成记录页
- 实现历史详情查看
- 实现个人设置页
- 将历史记录和用户设置持久化到 Supabase

完成标志：

- 用户能查看自己的历史数据
- 用户能维护自己的设置

### Phase 6：Supabase Auth 与权限收敛

目标：

- 统一用户身份、角色与数据权限

重点工作：

- 评估并推进从当前 JWT 方案迁移到 Supabase Auth
- 建立用户 profile / settings / history 的归属关系
- 使用角色字段与访问策略约束用户数据
- 落地 Row Level Security 策略

完成标志：

- 用户数据访问边界清晰
- 海报库、历史记录、设置具备可控权限策略

### Phase 7：生成能力强化

目标：

- 强化工作区真实生成流程

重点工作：

- 增加生成任务状态
- 增加 `AI Draw` 九维度的数据结构
- 增加参考海报到参数的识别映射逻辑
- 增加生成结果记录与结果展示
- 明确哪些逻辑直接调用 Supabase，哪些逻辑通过 Express 编排层执行

完成标志：

- 生成任务具备完整状态流
- `AI Draw` 具备基础参数灌入能力

### Phase 8：Express 编排层收敛

目标：

- 让 Express 从“当前 starter 后端”转变为“业务编排层”

重点工作：

- 保留 AI 模型调用、密钥保护、复杂工作流编排等服务端职责
- 移除不必要的重复 CRUD
- 保持与 Supabase 的职责边界清晰

完成标志：

- 数据主存储在 Supabase
- Express 只承担必须放在服务端的逻辑

### Phase 9：工程质量与上线准备

目标：

- 把产品从原型推进到可稳定迭代状态

重点工作：

- 测试体系
- 错误处理
- 日志与监控
- 部署与环境配置
- 数据迁移与环境分层

完成标志：

- 具备稳定的开发、测试、上线流程

## 四、关键技术专题路线

### 1. Supabase 数据底座

优先级：

- 极高

原因：

- 这是后台数据、海报库管理和用户数据管理的统一底座

### 2. 路由与导航

优先级：

- 高

原因：

- 所有页面级功能都依赖清晰路由结构

### 3. 轻量状态机

优先级：

- 高

原因：

- 海报库、工作区、模式切换、生成提交都需要清晰状态表达

建议：

- 先使用 React state/reducer + URL query
- 只有在复杂度明显上升后，再考虑引入专门状态机工具

### 4. 数据模型

优先级：

- 高

原因：

- 海报、标签、生成记录、设置与用户数据都依赖结构化数据

### 5. Supabase Storage

优先级：

- 中高

原因：

- 海报资源、生成结果、用户素材都需要统一文件存储路径

### 6. 生成引擎与 AI 能力接入

优先级：

- 中高

原因：

- 这是产品核心价值，但应建立在页面与数据基线稳定之后

### 7. 自定义后台管理页

优先级：

- 中低

原因：

- 当前先由 Supabase Studio 承担后台数据维护工作
- 只有当运营效率不足时，再考虑建设应用内后台

## 五、推荐实施顺序

建议的真实落地顺序是：

1. Supabase 项目与数据结构
2. 应用骨架与路由
3. 生成工作区基础版
4. 海报库页闭环
5. 历史记录与个人设置
6. Supabase Auth 与权限策略
7. 生成能力强化
8. Express 编排层收敛
9. 测试、部署与上线准备

## 六、当前不建议过早投入的事项

- 一开始就建设完整自定义后台管理系统
- 一开始就引入复杂工作流引擎
- 一开始就上重型状态机框架
- 一开始就做个性化推荐系统
- 一开始就做社区内容生态
- 在 Supabase 数据底座还没稳定前就做大规模基础设施重构
