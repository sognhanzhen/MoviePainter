# MoviePainter 项目宪法

本文档是本仓库中所有 AI 代理、自动化流程与协作者共同遵守的最高层工作宪法。

## 宪法优先级

优先级顺序如下：

1. 当前对话中用户明确表达的决策
2. 本 `AGENT.md`
3. vendored 在 `.claude/skills/gstack/` 中的上游 gstack 参考文件
4. 本地模块文档，例如 `README.md`、未来的架构文档，以及代码内注释

如果两条规则冲突，以优先级更高者为准。如果冲突会影响重要决策，必须先停下来对齐，再继续实现。

## 项目定义

MoviePainter 是一个电影海报生成 Web 产品。目标系统必须包含：

- 面向未登录用户的营销落地页
- 用户登录模块
- 海报库模块
- 生成工作区模块
- 用户个人模块
- 后台数据与内容管理能力

项目的长期目标不只是把产品做出来，还要让整个项目具备自动化能力，使未来不同业务场景可以调用不同的 gstack 工作流、工具或编排路径。

## gstack 接入状态

`gstack` 已作为 vendored 依赖引入到本项目中，路径如下：

- `.claude/skills/gstack/`

该 vendored 副本来自以下上游仓库快照：

- `https://github.com/garrytan/gstack`

规则如下：

- 不得对 `.claude/skills/gstack/` 做裁剪、局部复制或选择性删除。
- 必须将该 vendored gstack 目录视为本项目完整的上游参考基线。
- 如果需要刷新 gstack，优先使用干净的上游同步方式，不要做零散的手工挑拣。
- 项目特有规则应写在本 `AGENT.md` 中，而不是直接修改上游 gstack 的理念文件，除非项目明确决定要分叉。

## gstack 在本项目中的含义

在 MoviePainter 中，gstack 首先被采用为项目的规划、评审、QA 与交付操作系统。

默认解释如下：

- gstack 当前是研发工作流层。
- 除非经过明确设计，否则 gstack 不自动等同于产品运行时核心。
- 未来不同业务场景可能会调用不同的 gstack 能力，但这些运行时或后台编排集成必须经过有意设计，不能默认隐含成立。

## 强制工作流

对于任何非简单级别的功能、模块、架构变更、工作流变更或自动化变更，默认顺序如下：

1. `/office-hours`
2. `/plan-ceo-review`
3. `/plan-eng-review`
4. 如果面向终端用户，则执行 `/plan-design-review`
5. 如果面向开发者、运营人员或内部管理员，则执行 `/plan-devex-review`
6. 实现
7. `/review`
8. `/qa` 或 `/qa-only`
9. `/ship`
10. `/document-release`
11. `/retro`

如果任务同时跨越产品、设计、架构与开发体验，优先使用：

- `/autoplan`

## 编码前强制对齐

除非用户明确豁免，否则对战略性工作不得直接跳入实现。

战略性工作包括：

- 新的顶层模块
- 重大工作流变化
- 数据模型变化
- 认证与权限模型变化
- 后台系统变化
- 自动化与编排变化
- 第三方平台集成变化

默认规则如下：

- 对战略性工作，编码前先进行 3-5 轮对齐。
- 在存在关键取舍时，每次只问一个高杠杆问题。
- 对齐结束后，在开始实现前先总结已确认决策、非目标项以及未解决风险。

这条规则对当前的 MoviePainter 尤其重要，因为产品基础结构仍在成型阶段。

## 本项目采用的 gstack 核心理念

### 1. Boil the Lake

当 AI 把完整实现的边际成本降得很低时，只要问题规模仍然是“lake”而不是“ocean”，就优先选择完整且可维护的版本，而不是走捷径。

应用到 MoviePainter 上：

- 认证功能不能只有登录，还必须包含受保护路由、会话失效处理与退出登录行为。
- 海报生成功能不能没有任务状态、错误处理与可审计性。
- 后台工具不能没有权限控制、运营可见性与安全默认值。
- UI 流程不能缺少空态、加载态、错误态与成功态。

### 2. Search Before Building

在构建不熟悉的基础设施、工作流模式或集成之前：

- 先搜索现有仓库
- 再搜索上游 gstack 能力
- 再查阅第一手官方文档，再决定是否定制实现

不要重新发明一个仓库里已经存在、gstack 已经提供、或平台运行时已经内建的系统。

### 3. User Sovereignty

AI 提建议，用户做决定。

规则如下：

- 多模型一致只是信号，不是命令。
- 如果某个建议会改变用户既定方向，必须先提出并征求确认。
- 不得把战略决策偷偷夹带进实现过程。
- 只要用户已经表达明确偏好，就应当遵循，除非用户明确要求替代方案。

## MoviePainter 专属架构意图

产品应当演进为六个清晰的领域：

### 1. 营销落地页

范围：

- 未登录首页
- 产品定位表达
- 登录与开始创作的入口
- 产品价值说明
- 以转化为目标的页面布局

默认预期：

- 未登录用户不应直接落到应用壳层中。
- 落地页是营销与转化界面，不是占位页面。

### 2. 身份与访问控制

范围：

- 注册
- 登录
- 退出登录
- 会话校验
- 受保护路由
- 角色与权限模型
- 登录成功后跳转进入生成工作区

默认预期：

- 普通用户与管理员不能共用模糊的权限模型。
- 管理员权限必须明确且可审计。

### 3. 海报库

范围：

- 官方精选海报库
- 顶部筛选模块
- 不规则灵感排列，而不是统一卡片网格
- 海报资产记录
- 详情弹窗
- 元数据与推荐内容
- 进入 `AI Chat` 或 `AI Draw` 的模式选择跳转

默认预期：

- 第一阶段海报库来源是官方精选，而不是社区混合流。
- 点击海报必须先打开详情弹窗，而不是直接跳转。
- 点击 `Use` 后，必须先询问用户要进入 `AI Chat` 还是 `AI Draw`，再进入工作区。

### 4. 生成工作区

范围：

- 顶部区域标题为 `Start Creating With AI Chat/AI Draw`
- 默认模式为 `AI Chat`
- 上半区支持 `AI Chat` 与 `AI Draw` 双模式
- 下半区承载电影海报灵感与推荐内容
- 下半区采用不规则 masonry 风格布局，而不是标准网格
- 结果查看
- 重试与再细化循环
- 与模式相关的海报注入流程

默认预期：

- 生成是一个工作流，而不是单一按钮。
- 每个生成任务都应尽可能具备状态、参数、输出以及失败原因。

### 5. 用户个人模块

范围：

- 用户设置
- 个人管理
- 历史生成记录
- 生成详情查看

默认预期：

- 用户应能查看自己的历史生成记录，并进入详情查看。
- 个人历史记录是产品界面的一部分，而不仅仅是审计日志。

### 6. 后台与运营

范围：

- 用户管理
- 官方精选库维护
- 海报详情信息维护
- 参数标签维护
- 推荐内容维护
- 使用情况可视化
- 运营控制项

默认预期：

- 当前阶段后台数据、海报库内容与用户数据默认通过 Supabase 管理。
- 当前阶段不强制要求先建设应用内后台管理页。
- 如果未来需要自定义后台页，也不能与 Supabase 的数据源分叉。

## 已确认的交互基线

以下交互决策已经完成对齐，除非用户明确重新打开讨论，否则都应视为产品基线规则。

### 工作区模式系统

- 可见标题为 `Start Creating With AI Chat/AI Draw`。
- 默认模式为 `AI Chat`。
- `AI Chat` 与 `AI Draw` 是同一个生成工作区中的平级模式。

### AI Chat 模式

- 展示 AI 对话界面。
- 用户点击灵感海报后，必须先显示海报详情弹窗。
- 在海报详情中，若用户选择 `AI Chat`，点击 `Use` 后应进入 `AI Chat` 流程。
- 进入 `AI Chat` 后，所选海报应被插入到对话上下文中。

### AI Draw 模式

- 展示六个预设参数模块：
  - `角色`
  - `风格`
  - `氛围`
  - `色调`
  - `构图`
  - `比例`
- 模块支持用户主动选择与权重调节。
- 用户点击灵感海报后，必须先显示海报详情弹窗。
- 在海报详情中，若用户选择 `AI Draw`，点击 `Use` 后应进入 `AI Draw` 流程。
- 进入 `AI Draw` 后，工作区应先展示海报插入占位，而不是静默自动填满所有参数。
- 随后由用户多选六个参数模块中哪些需要接收参考海报的值。
- AI 负责执行图像理解，提取海报属性，并且只填充用户选中的模块。

### 海报库跳转流程

- 点击海报 -> 打开详情弹窗
- 在详情弹窗点击 `Use` -> 选择 `AI Chat` 或 `AI Draw`
- 选择模式 -> 进入目标工作区模式
- 目标工作区执行对应模式的注入逻辑

## 未来业务场景自动化规则

MoviePainter 预期未来支持不同业务场景。每一个场景级自动化都必须定义：

- 触发条件
- 目标执行角色
- 输入
- 输出
- 成功标准
- 人工审批边界
- 失败回退策略
- 可观测性或审计轨迹

任何场景自动化都不能以隐藏的一次性逻辑实现。它必须映射到明确的工作流。

推荐形态如下：

- 场景注册表
- 可复用工作流步骤
- 可配置的 prompts 或 presets
- 明确的策略与权限边界
- 可观察的执行历史

## 本仓库工程规则

- 在范围、架构与 UI 职责明确之前，不实现重大模块。
- 新增任何数据实体前，必须先明确其归属、生命周期与后台可见性要求。
- 任何异步生成流程都必须先有状态建模。
- 任何上传或资产流程都必须有校验、失败态与存储归属说明。
- 任何后台功能都必须有角色权限控制。
- 后台工具不得存在静默的破坏性行为。
- 任何 UI 页面都必须有空态、加载态、成功态与错误态。
- 任何“AI 魔法”表述背后都必须有清晰可解释的实际工作流。
